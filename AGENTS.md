# AGENTS

Постоянные правила работы для Codex в проекте `INFINDA`.

## 1. Общий принцип

Любая реализация должна быть:
- понятной;
- модульной;
- расширяемой;
- проверяемой;
- без хаоса в структуре;
- без дублирования;
- без скрытой магии;
- без временных костылей, если они не согласованы.

Codex не должен писать код "как получится". Любая работа должна выглядеть как инженерный проект, который можно поддерживать, расширять, тестировать и передавать другому разработчику.

## 2. Обязательный порядок работы перед кодом

Перед написанием кода Codex обязан:
1. Изучить текущую структуру проекта.
2. Прочитать `README.md`, `AGENTS.md`, `docs/RULES.md`, `docs/MEMORY.md`.
3. Понять, какие HTML-макеты являются визуальным эталоном.
4. Составить короткий план реализации.
5. Указать, какие файлы будут изменены или созданы.
6. Задать вопросы, если есть архитектурная неопределенность.
7. Только после этого приступать к реализации.

Дополнительно:
1. Не начинать реализацию сразу без изучения контекста.
2. Не менять стек без согласования.
3. Если задача большая, делить ее на этапы и выполнять только первый согласованный этап.
4. Не принимать крупные технические решения без согласования.

## 3. Работа с HTML-макетами

1. Не переносить HTML-макеты хаотично.
2. HTML в корне проекта считать визуальным эталоном.
3. Не ломать существующий визуальный стиль без отдельного согласования.
4. Перед переносом макета в приложение сначала определить:
   - какие блоки станут layout;
   - какие блоки станут компонентами;
   - какие данные будут статическими;
   - какие данные позже будут приходить из API.
5. Нельзя переносить HTML один в один в грязный JSX/TSX без разделения на компоненты.

## 4. Структура проекта

Нужно соблюдать модульную структуру:
- `apps/web`
- `apps/api`
- `packages/shared`
- `tests`
- `infra`
- `docs`

Запрещено:
- создавать файлы с неясным назначением;
- смешивать временный код с основной реализацией;
- оставлять неиспользуемые функции, импорты, переменные и файлы;
- делать быстрое решение, которое потом невозможно поддерживать.

## 5. Архитектурные требования

Код должен быть разделен по ответственности.

Запрещено смешивать в одном месте:
- UI;
- бизнес-логику;
- работу с API;
- работу с базой данных;
- валидацию;
- конфигурацию;
- тесты;
- инфраструктурные настройки.

Каждый слой должен иметь понятную зону ответственности.

Общие правила:
1. Не создавать монолитные файлы.
2. Не создавать один огромный компонент на всю страницу.
3. Не добавлять новую логику поверх старой, если старая уже требует рефакторинга.
4. Не дублировать код.
5. Не усложнять архитектуру без причины.
6. Не добавлять временные костыли без явной пометки `TODO` и плана удаления.
7. Новую логику не смешивать со старой, если старую сначала нужно вынести, очистить или переорганизовать.

Если существующая структура мешает правильной реализации, Codex должен сначала предложить рефакторинг, объяснить причину и дождаться согласования.

## 6. Правила именования и читаемости

1. Писать понятные имена файлов, функций и компонентов.
2. Делать структуру понятной без лишнего шума и хаоса.
3. Предпочитать небольшие понятные модули перегруженным файлам.
4. Все новые сущности и решения должны называться ясно и единообразно.

Запрещены неясные названия:
- `data`
- `item`
- `temp`
- `test1`
- `newFile`
- `helper`
- `utils2`
- `final`
- `copy`
- `fix`
- `stuff`

Исключение:
- короткие локальные переменные в маленьком очевидном контексте.

## 7. Требования к frontend-коду

Frontend должен быть организован аккуратно.

Обязательные правила:
1. Компоненты должны быть небольшими.
2. Повторяющиеся UI-блоки нужно выносить в отдельные компоненты.
3. Страницы не должны превращаться в огромные файлы.
4. Логика работы с API не должна находиться прямо внутри сложных UI-компонентов.
5. Типы данных должны быть описаны отдельно.
6. Названия компонентов должны быть понятными.
7. Визуальный стиль HTML-макетов нельзя ломать без согласования.
8. Нельзя добавлять лишние UI-библиотеки без объяснения причины.
9. Константы не должны быть разбросаны по файлам.
10. Стили должны быть организованы единообразно.

Рекомендуемая логика разделения:
- `pages` или `app` — страницы и маршруты;
- `components` — переиспользуемые компоненты;
- `features` — функциональные модули;
- `shared` — общие компоненты, утилиты, типы;
- `services` или `api` — работа с API;
- `styles` — глобальные стили, темы, переменные.

## 8. Требования к backend-коду

Backend должен быть организован по слоям.

Обязательные правила:
1. API-эндпоинты не должны содержать всю бизнес-логику.
2. Бизнес-логика должна находиться в сервисах.
3. Работа с базой данных должна быть отделена от API-слоя.
4. Валидация входных данных должна быть явной.
5. Ошибки должны обрабатываться предсказуемо.
6. Конфигурация не должна быть захардкожена в коде.
7. Секреты нельзя хранить в репозитории.
8. Для важных сценариев должны быть тесты.
9. Миграции базы данных должны быть осознанными и проверяемыми.
10. Интеграции с внешними сервисами должны быть изолированы.

Рекомендуемая логика разделения:
- `models` — модели данных;
- `schemas` или `serializers` — входные и выходные структуры;
- `views`, `controllers` или `routes` — API-слой;
- `services` — бизнес-логика;
- `repositories` или `managers` — доступ к данным;
- `integrations` — внешние сервисы;
- `tasks` — фоновые задачи;
- `config` или `settings` — настройки;
- `tests` — тесты.

## 9. Требования к API

API должен быть понятным и стабильным.

Обязательные правила:
1. Названия эндпоинтов должны быть логичными.
2. Ответы API должны иметь предсказуемую структуру.
3. Ошибки должны возвращаться в едином формате.
4. Статусы HTTP должны использоваться корректно.
5. Нельзя возвращать лишние внутренние данные.
6. Нельзя раскрывать секреты, токены, пароли, внутренние ошибки.
7. Для новых API нужно описывать:
   - метод;
   - URL;
   - входные данные;
   - успешный ответ;
   - ошибки;
   - права доступа.

Пример структуры ошибки:

```json
{
  "error": {
    "code": "DEVICE_LIMIT_EXCEEDED",
    "message": "Device limit exceeded",
    "details": {}
  }
}
```

## 10. Требования к базе данных

Codex не должен менять структуру базы данных хаотично.

Перед изменением моделей или схемы БД нужно:
1. Объяснить, зачем нужно изменение.
2. Описать, какие таблицы или поля добавляются.
3. Проверить, не дублирует ли новая сущность уже существующую.
4. Создать миграцию.
5. Указать, как применить миграцию.
6. Указать риски для существующих данных.

Запрещено:
- удалять поля без согласования;
- переименовывать таблицы без плана миграции;
- хранить JSON там, где нужна нормальная структура, без причины;
- хранить пароли, токены и ключи в открытом виде;
- создавать связи без индексов там, где ожидаются частые запросы.

## 11. Требования к безопасности

Безопасность должна учитываться по умолчанию.

Запрещено:
- хранить секреты в коде;
- коммитить `.env` с реальными данными;
- логировать пароли, токены, cookies, приватные ключи;
- отключать авторизацию ради удобства;
- делать публичные endpoints без необходимости;
- использовать `eval`;
- отключать CSRF, CORS или security-настройки без объяснения;
- возвращать пользователю внутренние stack trace;
- доверять входным данным без валидации.

Обязательно:
- использовать переменные окружения;
- валидировать входные данные;
- проверять права доступа;
- ограничивать доступ к административным функциям;
- документировать чувствительные места;
- добавлять безопасные значения по умолчанию.

## 12. Требования к тестам

Любая значимая логика должна иметь тесты.

Минимально проверять:
- успешный сценарий;
- ошибочный сценарий;
- пустые или некорректные данные;
- права доступа;
- граничные случаи;
- работу важных сервисов;
- работу API, если добавлен новый endpoint.

Если тесты временно не добавлены, Codex обязан явно написать:
- почему тесты не добавлены;
- какие тесты нужно добавить;
- где они должны лежать;
- как их запускать.

Нельзя писать, что все работает, если проверки не запускались.

## 13. Проверки перед завершением задачи

После реализации Codex обязан выполнить доступные проверки.

Для frontend:
- установка зависимостей, если нужна;
- форматирование;
- линтер;
- type-check;
- сборка;
- тесты;
- ручная проверка страницы, если возможно.

Для backend:
- форматирование;
- линтер;
- тесты;
- миграции;
- запуск сервера, если возможно;
- проверка основных endpoints;
- проверка ошибок.

Если команда не может быть выполнена, Codex обязан написать причину.

## 14. Требования к документации

Если меняется архитектура, структура, API, сценарии запуска или переменные окружения, документация должна быть обновлена.

Нужно обновлять:
- `README.md`;
- `docs/MEMORY.md`;
- `docs/RULES.md`;
- документацию конкретного модуля;
- API-документацию;
- `.env.example`, если добавлены новые переменные.

Запрещено менять поведение проекта и не фиксировать это в документации.

## 15. Работа с legacy-кодом

Если в проекте уже есть старая логика, Codex обязан:
1. Сначала понять, зачем она нужна.
2. Проверить, используется ли она.
3. Не писать новую логику поверх старой без анализа.
4. Если нужно заменить старую логику, предложить план миграции.
5. Если нужно удалить старую логику, объяснить, почему это безопасно.
6. Если есть риск поломки, указать его явно.

Запрещено:
- молча удалять старый код;
- оставлять две конфликтующие реализации;
- копировать старую логику в новое место без причины;
- делать временную вторую систему рядом с первой.

## 16. Работа с зависимостями

Нельзя добавлять новые библиотеки без причины.

Перед добавлением зависимости нужно объяснить:
1. Зачем она нужна.
2. Почему нельзя решить без нее.
3. Насколько она критична.
4. Где она будет использоваться.
5. Есть ли риски.

Запрещено:
- добавлять тяжелые библиотеки ради одной простой функции;
- добавлять несколько библиотек с одинаковым назначением;
- добавлять устаревшие или сомнительные пакеты;
- менять major-версии без согласования;
- обновлять весь dependency tree без необходимости.

## 17. Работа с ошибками

Ошибки должны обрабатываться явно.

Запрещено:
- глушить ошибки пустым `catch`;
- писать `try/catch` без нормальной обработки;
- показывать пользователю внутренние ошибки;
- логировать чувствительные данные;
- возвращать непонятное "Something went wrong" без кода ошибки для API.

Правильно:
- логировать техническую ошибку безопасно;
- пользователю возвращать понятное сообщение;
- для API использовать единый формат ошибки;
- для UI показывать аккуратное состояние ошибки.

## 18. Производительность

Нужно избегать решений, которые очевидно ухудшают производительность.

Запрещено:
- делать лишние запросы к API;
- делать тяжелые вычисления прямо в UI без необходимости;
- загружать огромные данные без пагинации;
- делать N+1 запросы на backend;
- хранить большие временные данные в неподходящем месте;
- ререндерить большие компоненты без причины.

Если есть риск производительности, его нужно указывать в отчете.

## 19. Логи и диагностика

Логи должны помогать диагностике, но не раскрывать секреты.

Обязательно логировать:
- важные системные ошибки;
- ошибки интеграций;
- проблемы с платежами;
- проблемы с авторизацией;
- проблемы с VPN-нодами;
- критичные фоновые задачи.

Запрещено логировать:
- пароли;
- токены;
- cookies;
- приватные ключи;
- полные персональные данные без необходимости;
- платежные секреты.

## 20. Правила внесения изменений

1. Перед изменением существующего кода сначала объяснить, почему его нужно менять.
2. Реализовывать задачу маленькими этапами.
3. После каждого этапа обновлять документацию, если изменились структура, правила, архитектура или сценарии.
4. Если есть архитектурная неопределенность, сначала задавать вопросы, потом писать код.

## 21. Финальный отчет

После каждой реализации Codex обязан дать отчет:
1. Что было сделано.
2. Какие файлы созданы.
3. Какие файлы изменены.
4. Какие проверки выполнены.
5. Какие проверки не выполнены и почему.
6. Какие риски остались.
7. Что нужно сделать следующим шагом.
8. Какие вопросы требуют согласования.

Рекомендуемый формат:

```text
Итог:

Сделано:
- ...

Измененные файлы:
- ...

Проверки:
- ...

Не проверено:
- ...

Риски:
- ...

Следующий шаг:
- ...
```

## 22. Главный запрет

Codex не должен делать вид, что задача полностью готова, если:
- тесты не запускались;
- сборка не проверялась;
- есть ошибки;
- есть нерешенные вопросы;
- часть функционала сделана предположительно;
- документация не обновлена;
- решение временное.

В таких случаях нужно честно писать статус:

`Реализовано частично. Требуется проверка, согласование или доработка.`
