# PROJECT REVIEW REPORT

Дата ревью: `2026-06-06`

## 1. Короткий вывод

Проект в целом собран нормально:
- структура по папкам понятная;
- разделение на `apps/web`, `apps/api`, `packages/shared`, `docs`, `tests`, `infra` соблюдено;
- frontend и backend уже не выглядят как хаотичный прототип;
- основные проверки проходят.

Критических развалов по архитектуре не найдено.

Главная проблема проекта сейчас не в том, что он "плохо написан", а в том, что:
- часть обзорной документации нужно постоянно держать синхронной с кодом;
- корневой слой `tests/` только начал оформляться и пока еще не наполнен реальными сценариями;
- `infra` пока остается пустым каркасом;
- есть отдельные точечные дефекты в прикладной логике, которые нужно закрывать по мере ревью.

## 2. Что было проверено

Проверены:
- структура каталогов проекта;
- backend-модули `auth`, `profile`, `devices`, `subscription`, `support`, `telegram`, `activity`, `health`;
- frontend-слой `apps/web`, особенно `auth`, `cabinet`, BFF/proxy route handlers;
- пакет `packages/shared`;
- основная документация в `README.md` и `docs/*`;
- доступные автоматические проверки.

## 3. Оценка структуры проекта

### Что хорошо

- `apps/web` и `apps/api` разделены правильно.
- В `apps/web` маршруты в `src/app` в основном остаются тонкими.
- Основная UI-логика вынесена в `features` и `shared`.
- В `apps/api` домены разнесены по приложениям, а не свалены в один файл.
- `packages/shared` используется по назначению: как слой контрактов между frontend и backend.
- Документация проекта лежит в ожидаемых местах и в целом читается понятно.

### Что пока не идеально

- `tests/` уже разложен на `tests/api` и `tests/web`, но там пока только каркас без реальных сценариев.
- `infra/` пока только заготовка.
- В рабочем окружении остаются generated-каталоги вроде `.next`, `.venv`, `__pycache__`, `node_modules`, но они gitignored и не являются структурной ошибкой репозитория.

## 4. Что исправлено в рамках этого ревью

### Исправление 1. Профиль и email

Было:
- пользователь мог обновить email без проверки уникальности;
- `username` не синхронизировался с новым email;
- это создавало риск дублей и рассинхрона идентичности пользователя.

Стало:
- при обновлении профиля email теперь валидируется на уникальность;
- email нормализуется;
- `username` синхронизируется с новым email;
- добавлены backend-тесты на этот сценарий.

### Исправление 2. Ошибки регистрации на frontend

Было:
- backend мог вернуть ошибку по полю `name`;
- frontend не показывал эту ошибку пользователю.

Стало:
- ошибка `name` теперь правильно извлекается;
- поле `Имя` в форме регистрации умеет показывать backend-ошибку.

### Исправление 3. Документация по структуре

Было:
- `docs/PROJECT_STRUCTURE.md` отставал от реального состояния backend и shared-слоя.

Стало:
- документ синхронизирован с текущими доменами и контрактами.

### Исправление 4. BFF и недоступный backend

Было:
- если Next BFF не мог достучаться до backend по сети, route handler мог упасть необработанной ошибкой.

Стало:
- BFF теперь возвращает предсказуемую ошибку `502 UPSTREAM_UNAVAILABLE`;
- это делает поведение frontend стабильнее при локальном падении backend или сетевой проблеме.

### Исправление 5. Базовый CI и каркас repo-level tests

Было:
- автоматического CI в репозитории не было;
- корневой `tests/` был почти пустым.

Стало:
- добавлен базовый GitHub Actions workflow для backend и frontend;
- оформлены `tests/api` и `tests/web` как отдельные слои со своим назначением.

## 8. Общая оценка проекта

Сейчас проект выглядит как:
- хороший рабочий каркас продукта;
- с уже реальной бизнес-логикой, а не только с моками;
- без критического архитектурного хаоса;
- но еще без полноценного слоя системных тестов и инфраструктурного контура.

Простыми словами:

Проект уже можно понимать, поддерживать и развивать.
Главное сейчас:
- не разъезжаться между кодом и документацией;
- не откладывать интеграционные тесты слишком надолго;
- не превращать `cabinet` и support-сценарии в новый монолит по мере роста функционала.

## 9. Рекомендованный следующий шаг

Лучший следующий шаг после этого ревью:

1. Поднять первый реальный слой `tests/api` на уровне нескольких доменов сразу.
2. После этого уже развивать `infra` и deployment-процессы.

Это даст проекту больше пользы, чем случайная “дополнительная чистка” без нового покрытия и без новых гарантий.
