# MEMORY

Назначение:
- Этот файл хранит рабочий контекст проекта, чтобы не терять договоренности между этапами.

Текущий контекст:
- Проект `INFINDA` начинается с набора HTML-макетов.
- Визуальные макеты находятся в каталоге `шаблоны и работа/`.
- `apps/web` на `Next.js` и `apps/api` на `Django + DRF` уже инициализированы.
- Текущий технический статус: базовый каркас уже пройден, собраны основные frontend-маршруты, BFF/proxy-слой и несколько backend-доменов с тестами.
- Для frontend уже вынесен общий layout-слой с `header` и `footer`, чтобы изменения этих блоков применялись ко всем страницам сразу.
- Для frontend уже добавлен общий механизм reveal-on-scroll для анимации блоков при прокрутке.
- Для frontend уже создан общий стилевой слой `shared/styles/site-primitives.module.css` для контейнеров, фоновой анимации и базовых non-selectable правил.
- Для frontend уже добавлен общий mobile-navigation паттерн для header на маленьких экранах.
- Для frontend уже настроена реальная маршрутизация между `/` и `/features`, а также активное состояние header-навигации.
- Для frontend уже добавлен общий FAQ-аккордеон в `shared/ui`.
- В `apps/web` уже перенесены и выровнены по одному шаблону страницы `главная`, `возможности`, `цены`, `ресурсы`, `о нас`, `авторизация` и `личный кабинет`.
- Для `личного кабинета` уже используется отдельный dashboard-layout с более узкой рабочей областью и сворачиваемым боковым меню.
- Для `личного кабинета` уже зафиксирован единый UI-паттерн: sticky/collapsible sidebar, page-top actions, overlay-модалки для продления и профиля, accordion-список устройств и full-width support-view.
- Для ошибок сайта теперь используется отдельный модуль `error-page` с отдельными маршрутами под `403 / 404 / 500 / 502 / 503 / 505`, а не одна demo-страница со switcher.
- В backend начат первый доменный слой `auth`: отдельный модуль с `login/logout/me` на `Django + DRF` и token-based локальной авторизацией для стыковки с frontend `/auth`.
- В backend auth-слой уже расширен до `register/login/logout/me`, а frontend-форма регистрации больше не является только визуальной оболочкой.
- При регистрации новый пользователь теперь должен автоматически получать trial-подписку `Триал 3 дня` и стартовые маршруты, чтобы `/cabinet` не открывался в полупустом состоянии.
- Кабинет должен уметь корректно жить и без подписки: subscription API теперь должен описывать состояния `none / trial / active / expired`, а не ломать UI через `404`.
- Для paid-сценария теперь используется единый реальный платежный контур через `Platega SBP`: checkout создается на backend, а подтверждение идет через webhook и только потом активирует подписку.
- В backend уже добавлены модули `profile` и `devices` с реальными endpoint-ами `GET/PATCH /api/profile/me/`, `GET /api/devices/`, `POST /api/devices/<id>/revoke/`.
- В backend уже добавлен модуль `subscription` с реальным endpoint-ом `GET /api/subscription/`; вкладка `Подписка` в кабинете больше не живет только на моках.
- В backend уже добавлен модуль `activity` для хранения административного журнала действий пользователя: вход, выход, обновление профиля и отзыв устройства.
- `apps/web` больше не использует demo-login на `/auth`: frontend auth-форма теперь ходит в собственные Next route handlers `/api/auth/login|me|logout`, а те проксируют запросы в локальный `Django + DRF` backend.
- Токен и данные пользователя теперь сохраняются локально на стороне frontend: `localStorage` при `Запомнить меня` и `sessionStorage` без этой опции.
- `apps/web` уже подключен к backend-модулям `profile/devices` внутри `cabinet`: профиль обновляется через API, список устройств и отзыв устройства больше не держатся только на моках.
- В `apps/web` маршруты в `src/app` выступают как тонкие route-entry файлы, а основная UI-структура и контент разделены по feature-модулям в `src/features`.
- В `apps/web` уже есть route handlers для `auth`, `profile`, `devices`, `subscription`, чтобы браузер ходил не напрямую в Django, а через собственный frontend-слой.
- В `packages/shared` уже вынесен первый реальный контрактный слой: endpoint-paths, transport DTO и единый error-format для `auth/profile/devices/subscription`.
- Для `subscription` теперь есть отдельная модель платежной сессии `SubscriptionPayment`, чтобы хранить `checkout_url`, `external_payment_id`, статусы провайдера и обеспечивать идемпотентность webhook-подтверждения.
- Корневой `.env` проекта теперь должен оставаться чистым и содержать только актуальные переменные `INFINDA`; старый env от `Amonora` вынесен в `шаблоны и работа/amonora.legacy.env` как reference-источник.
- Корневой `tests` пока все еще в основном пустой: системный и интеграционный слой поверх приложений еще не развернут.
- Для локальной проверки уже подготовлены demo-данные кабинета: тестовый профиль, несколько устройств и demo-подписка `12 месяцев (безлимит)` с 4 маршрутами по странам. Конкретные локальные учетные данные не фиксируются в документации проекта.
- `Django admin` переведен на русский интерфейс через `LANGUAGE_CODE=ru-ru`, русские названия моделей и настроенный `admin.site` header/title.
- В `Django admin` карточка пользователя теперь должна показывать ФИО, inline-профиль и последние действия, а устройства должны явно различаться по состоянию `Активно/Отозвано`.
- В списках admin для пользователей и устройств теперь важно показывать `id`, чтобы было легче ориентироваться на больших выборках.
- В `Django admin` теперь также доступны inline-история платежей пользователя, ручные действия по выдаче и снятию подписки, отзыв всех устройств пользователя, а в разделе платежей есть помесячная финансовая сводка и ручная обработка статусов оплаты.
- Для `Django admin` добавлен собственный dark-dashboard слой `INFINDA`, визуально вдохновленный `Black Dashboard`, но без переноса чужих `apps/pages`, `dyn_api`, `dyn_dt` и без смены текущей архитектуры проекта.
- Этот admin-слой уже включает не только главную dashboard-страницу, но и branded login-страницу, быстрые ссылки на ключевые административные разделы и карточное представление списка приложений/моделей.
- Для admin-слоя теперь также настроены обе темы `dark/light` с едиными CSS-переменными и общим переключателем темы на экране входа и внутри панели управления.
- Dashboard админки уже начал превращаться во внутреннюю систему управления: добавлены отдельные блоки для проблемных платежей, подписок с близким окончанием и контроля устройств.
- Login-экран админки дополнительно отполирован: собственный loader, show/hide password, switch темы и более аккуратное клиентское поведение формы до и после попытки входа.
- Левое меню админки уже переведено на более широкий collapsible sidebar с раскрывающимися секциями приложений и локальным запоминанием состояния скрытия.

Что уже решено:
- Начинаем с этапа 1: каркас и планировка.
- Сразу ведем документацию внутри репозитория.
- Единый источник постоянных правил проекта, требований к качеству кода, архитектуре, проверкам и отчетности: `AGENTS.md`.
- Для новых задач используется единый шаблон планирования: `docs/TASK_PLAN_TEMPLATE.md`.
- Согласованный стек: `Next.js` для frontend и `Django + DRF` для backend.
- Код должен быть чистым, структура понятной, без сваливания логики в один большой слой.
- Работа идет через планирование и согласование.
- Тестирование является обязательной частью реализации.

Что важно помнить дальше:
- Перед внедрением нового крупного блока нужно согласование с пользователем.
- HTML-макеты нельзя воспринимать как временный мусор: это референс по внешнему виду.
- Архитектура должна быть модульной и расширяемой.
- Нужно избегать скрытых решений без фиксации в документации.
- Постоянные правила не дублируются по нескольким файлам: для них используется `AGENTS.md`.
- Перед любым новым кодом нужно проходить через порядок: изучение контекста, план, список файлов, вопросы при неопределенности, потом реализация.
- Документацию нужно держать синхронной с реальным кодом: часть старых описаний этапа foundation уже не отражает фактическую глубину реализации.
- Общие контракты уже начали выноситься в `packages/shared`, но backend пока использует их не как runtime-зависимость, а как согласованный reference-слой, закрепленный тестами.
